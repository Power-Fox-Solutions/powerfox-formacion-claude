Arquitectura y modelo de datos

  • Power Apps Model-Driven sigue la estructura de Dataverse; Canvas da control total de diseño.
  • Dataverse es la capa de datos compartida por todo el ecosistema.
  • Las tablas tienen relaciones padre-hija con herencia de Business Unit.
  • FLS (Field Level Security) controla visibilidad a nivel de campo, no solo de registro.
  • Los roles de seguridad funcionan en capas: BU, Teams y privilegios de tabla.

Modelo de seguridad

  • Business Units definen el scope de datos visible para cada usuario.
  • Teams permiten compartir registros cruzando jerarquías de BU.
  • Cada tabla tiene 4 niveles de acceso: User, BU, Parent BU, Organization.
  • Los 10 roles base del sistema tienen matrices de permisos documentables con Claude Code.

Clave: Separar la lógica de negocio de la seguridad. Mezclarlas complica el mantenimiento a medio plazo.

Conceptos clave para consolidar

Modelo de datos Dataverse

  • Tablas estándar vs tablas personalizadas (pfx_ prefix)
  • Tipos de relaciones: 1:N, N:N, Lookup
  • Herencia de BU en registros hijo
  • Calculated fields vs Rollup fields
  • Alternate keys para integraciones externas

Seguridad en capas

  • Roles: conjuntos de privilegios sobre entidades
  • Business Units: jerarquía organizativa que define scope
  • Teams: grupos para compartir registros cruzando BUs
  • FLS: Field Level Security para campos sensibles
  • Column Security Profiles: configuración de FLS por rol

El workflow Git que debo dominar

# Flujo completo de trabajo git switch main git pull git switch -c feature/mi-tarea # ... hago cambios ... git add archivo-especifico.ext git commit -m "feat(area): descripción corta" git push -u origin HEAD gh pr create --fill

Los prefijos de Conventional Commits:

refactor:Reorganizar sin cambiar comportamiento
