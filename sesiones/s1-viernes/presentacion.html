01

¿Qué es JavaScript en Model-Driven Apps?

20 min

💡 La idea

JavaScript en MDA es el código que se ejecuta en el navegador del usuario mientras usa la aplicación. Permite que el formulario reaccione en tiempo real a lo que hace el usuario: cambiar un campo, ocultar una sección, validar datos, consultar otra tabla, mostrar un mensaje.

Vive en Web Resources de tipo JavaScript dentro de una solución. Se asocia a formularios específicos y se dispara con eventos.

🎯 ¿Cuándo usar JavaScript?

  • Mostrar/ocultar campos o secciones según condiciones
  • Validar datos antes de guardar (ej: fecha fin > fecha inicio)
  • Calcular valores en tiempo real al cambiar un campo
  • Consultar datos de otras tablas y mostrarlos
  • Notificar al usuario con mensajes o alertas
  • Filtrar lookups dinámicamente según contexto

⚖️ JavaScript vs Business Rules vs Power Automate

NecesidadMejor herramientaPor qué
Ocultar campo si otro campo tiene valor XBusiness RuleSin código, visual, suficiente
Validación compleja con condiciones múltiplesJavaScriptLógica más flexible que BR
Consultar otra tabla al abrir el formularioJavaScript (Xrm.WebApi)BR no puede consultar datos
Enviar email al guardarPower AutomateMejor en server-side que client
Calcular campo siempre (UI + API)Plugin o Formula ColumnJS solo funciona en formulario
Filtrar lookup según otro campoJavaScriptSolo JS puede manipular PreSearch

Regla de oro: Si Business Rule puede hacerlo, usa Business Rule. Si se repite en muchos formularios y necesita complejidad, pásalo a JavaScript. Si necesita ejecutarse desde API o flujos, ya estás hablando de Plugin o Power Automate.

⚠️ JavaScript solo se ejecuta en el formulario. Si alguien crea/modifica un registro desde la API, un flow, o un import de datos, tu JavaScript NO se ejecuta. Si la lógica tiene que aplicarse siempre, necesitas Plugin o Power Automate.

Los 3 Eventos Principales + formContext

25 min · CLAVE

⚡ Los 3 eventos básicos

EventoCuándo se disparaUso típico
OnLoadAl abrir el formulario (crear o editar)Inicializar, ocultar campos según rol, pre-rellenar, validar permisos
OnChangeAl modificar un campo específico y salir de élCalcular, auto-rellenar otro campo, filtrar lookup, validar
OnSaveAl guardar (antes de guardar realmente)Validar datos, prevenir guardado si hay error

🎁 ¿Qué es formContext?

Es el "mando a distancia" del formulario. Con formContext puedes:

  • Leer o escribir valores de campos
  • Ocultar o mostrar campos, secciones y pestañas
  • Hacer campos obligatorios o de solo lectura
  • Mostrar notificaciones al usuario
  • Refrescar el formulario o un subgrid

⚠️ Xrm.Page está deprecado. Siempre usa formContext obtenido desde executionContext.

🔑 Patrón básico

function onLoad(executionContext) { // Paso 1: obtener formContext var formContext = executionContext.getFormContext(); // Paso 2: usar los métodos var nombre = formContext .getAttribute("pfx_name") .getValue(); if (nombre === null) { formContext.ui.setFormNotification( "Falta el nombre", "WARNING" ); } }

🛠️ Las 5 operaciones más comunes con formContext

// 1. Leer un valor var valor = formContext.getAttribute("pfx_campo").getValue(); // 2. Escribir un valor formContext.getAttribute("pfx_campo").setValue("Nuevo valor"); // 3. Ocultar o mostrar un campo formContext.getControl("pfx_campo").setVisible(false); // 4. Hacer obligatorio un campo formContext.getAttribute("pfx_campo").setRequiredLevel("required"); // 5. Mostrar una notificación formContext.ui.setFormNotification("Mensaje al usuario", "INFO", "id_notif");

💡 attribute = el dato del campo. control = la parte visual del campo en el formulario. Un mismo campo puede tener múltiples controles (si aparece en varias pestañas).

📝 Paso a paso detallado

  • 1. Crear Web Resource tipo JavaScript (.js) — nombre: pfx_/js/entityname.js
  • 2. Subir el archivo o pegar el código en el editor
  • 3. Abrir el formulario donde aplicará — Form Properties
  • 4. Libraries → Add Library → seleccionar tu JS
  • 5. Event Handlers → elegir evento (OnLoad / OnSave)
  • 6. Add → especificar la función (ej: LITOCLEAN.Projeto.onLoad)
  • 7. ✅ Marcar "Pass execution context as first parameter"
  • 8. Save + Publish

⚠️ Errores comunes al vincular

  • "Execution context is undefined" — olvidaste marcar el checkbox del paso 7
  • "Function X is not defined" — el nombre del namespace en Event Handler no coincide con el del JS
  • "Library not found" — subiste el web resource pero no lo publicaste
  • El código no se ejecuta — caché del navegador, Ctrl+F5 para forzar recarga
  • Cambios no aparecen — el web resource no se re-subió después de editarlo localmente

🏗️ Patrón que usamos en nuestro sistema

Todos los scripts del sistema siguen este patrón namespace:

if (typeof ProjectForm === "undefined") ProjectForm = { __namespace: true }; ProjectForm.Events = { OnLoad: function(executionContext) { /* ... */ }, OnSave: function(executionContext) { /* ... */ } }; ProjectForm.Functions = { CalculateTotals: function() { /* ... */ }, ApplyVisibilityRules: function() { /* ... */ } };

El Events contiene los handlers que se vinculan al formulario. Functions contiene la lógica reutilizable. Esta separación facilita mantenimiento.

Xrm.WebApi Básico — Leer Datos de Otras Tablas

20 min

🔌 ¿Qué es Xrm.WebApi?

Es la puerta para que tu JavaScript hable con Dataverse. Permite leer, crear, actualizar y eliminar registros desde el formulario.

Hoy vemos los dos métodos que más se usan:

  • retrieveRecord — Leer UN registro específico por su ID
  • retrieveMultipleRecords — Leer VARIOS registros con filtros

📖 retrieveRecord

// Leer un proyecto específico Xrm.WebApi.retrieveRecord( "pfx_project", "guid-del-registro", "?$select=pfx_project_code,pfx_project_name" ).then( function success(result) { console.log(result.pfx_project_name); }, function error(e) { console.log(e.message); } );

📚 retrieveMultipleRecords

// Leer actividades de un proyecto específico var projectId = "guid-del-proyecto"; var query = "?$select=pfx_activity_name,pfx_total_estimated_hours" + "&$filter=_pfx_project_id_value eq " + projectId + "&$orderby=pfx_activity_name asc"; Xrm.WebApi.retrieveMultipleRecords("pfx_project_activity", query).then( function success(result) { // result.entities es un array con todos los registros for (var i = 0; i < result.entities.length; i++) { console.log(result.entities[i].pfx_activity_name); } }, function error(e) { console.log(e.message); } );

Los 3 parámetros de la query: $select dice qué campos traer (menos es más rápido), $filter filtra con condiciones, $orderby ordena. Más opciones en la sesión avanzada.

⚠️ Gotcha importante: Para filtrar por un campo lookup, se usa _nombre_value (con underscore delante y _value detrás). No usas el nombre del campo directamente.

05

pfx_Common y pfx_Common_Helpers

15 min

🧰 ¿Qué son las librerías compartidas?

Son Web Resources de JavaScript que NO están vinculados a un formulario específico, sino que contienen funciones que todos los formularios pueden usar.

En vez de escribir el mismo código en 20 formularios, lo escribes una vez en una librería común y todos lo llaman.

📦 Las dos librerías del sistema

  • pfx_Common — Helpers genéricos que valen para cualquier formulario: leer/escribir campos, notificaciones, navegación, operaciones CRUD con Xrm.WebApi envueltas
  • pfx_Common_Helpers — Helpers específicos del negocio de Project Hub: cálculos de horas, conversión de divisas, lógica de totales

📦 Lo que hay dentro de pfx_Common

La librería expone varios namespaces con funciones listas para usar:

  • Common.Api — Retrieve, UpdateRecord, CreateRecord, RetrieveMultipleRecords, DeleteRecord (wrappers de Xrm.WebApi)
  • Common.FormContext — Init, GetValue, SetValue, TabVisible, SectionVisible, setFieldDisabled, AddFilterToLookup, setFormNotification
  • Common.GlobalContext — GetUserSettings, GetUserSecurityRoles

Y pfx_Common_Helpers contiene funciones específicas del negocio:

  • Common_Helpers.Custom_Action — Execute (llamadas a Custom API)
  • Common_Helpers.Custom_Functions — CalculateHoursTotalsSummary, CalculateAndConvertTotals, GetTotalSumHours

🎯 Cómo se usa desde un formulario

❌ Sin librería común

// Cada formulario repite esto var fc = executionContext.getFormContext(); var attr = fc.getAttribute("pfx_campo"); if (attr !== null) { var val = attr.getValue(); // ... usar val }

✅ Con pfx_Common

// Una línea, con null check incluido Common.FormContext.Init(executionContext); var val = Common.FormContext.GetValue("pfx_campo");

💡 Una librería compartida bien hecha reduce el código de cada formulario un 40-60% y elimina bugs repetidos en varios sitios.

Skills y Agents de Claude Code — En Vivo

45 min · BLOQUE ESTRELLA

🤖 Dos momentos, dos herramientas

Hay dos momentos muy distintos cuando trabajas con código:

1. Cuando escribes código nuevo → Las Skills se aplican automáticamente y hacen que el código nazca ya con las buenas prácticas.

2. Cuando revisas o mantienes código existente → Los Agents son especialistas que analizan y proponen mejoras concretas.

En los próximos 45 minutos vamos a crear una skill y dos agents en directo, y los aplicaremos a código real para ver la diferencia.

🎬 El punto de partida — Un código real

Este es un script de ejemplo de un formulario Account. Funciona, pero tiene oportunidades de mejora. ¿Cuáles detectas?

var LITOCLEAN = LITOCLEAN || {}; LITOCLEAN.Account = (function() { function onLoad(executionContext) { var formContext = executionContext.getFormContext(); _checkRating(formContext); } function _checkRating(formContext) { var avgRating = formContext.getAttribute("cr_mediacalificacion").getValue(); if (avgRating !== null && avgRating < 5) { // Notificación visual en el formulario formContext.ui.setFormNotification( "⚠️ La media de calificación de contactos es menor a 5", "WARNING", "low_rating" ); // Popup de alerta Xrm.Navigation.openAlertDialog({ text: "La calificación media de los contactos es " + avgRating + ". Se recomienda revisar.", title: "Advertencia de Calificación" }); } else { formContext.ui.clearFormNotification("low_rating"); } } return { onLoad: onLoad }; })();

Ejercicio en vivo: Antes de continuar, ¿qué mejoras veis? Toma 2 minutos para que el equipo las enumere. Vamos a ver si Claude Code detecta las mismas (y más).

Creando la Skill en vivo

🎨 Skill: MDA JavaScript Writer

Instrucciones que Claude Code aplicará SIEMPRE que genere JavaScript para MDA.

const/let, nunca var formContext, nunca Xrm.Page Namespace pattern Null checks async/await JSDoc

🎯 Cuándo se activa

Cada vez que pidas a Claude Code generar código JavaScript nuevo para un formulario Model-Driven, la skill se aplica automáticamente en background.

No tienes que recordarle las reglas en cada prompt. El código nace ya correcto.

🔄 Antes y después con la Skill activa

Una vez creada la skill, le pedimos a Claude Code que regenere el mismo script del Account. Observad las diferencias:

❌ Original (sin skill)

  • var en todas las declaraciones
  • Sin null check en getAttribute
  • Magic number 5 sin constante
  • Sin JSDoc
  • Mezcla notification + dialog sin criterio

✅ Con Skill (regenerado)

  • const por defecto, let solo si reasigna
  • Null check antes de .getValue()
  • const MIN_RATING = 5 al inicio
  • JSDoc con @param executionContext
  • Una única forma de notificar al usuario
