Contexto: dónde estamos

Recordatorio rápido para tu cabeza antes de empezar.

De dónde venimos

  • Sesiones 1–5 — repo montado, alumnos con su fork, han escrito su README.md, su CLAUDE.md y han generado el recap.html personalizado del curso.
  • Hasta ahora Claude solo ha generado contenido (markdown, html). Nunca código ejecutable.

A dónde vamos hoy

  • Hoy (miércoles, 3h): cada alumno construye una weather app dentro de su carpeta alumnos/[nombre]/weather-report/. Tres prompts encadenados, llegan a localhost:5173 con datos reales de Open-Meteo.
  • NO vamos a Vercel hoy. El clímax es ver la app funcionando en su pantalla. Subir a la web queda para una sesión futura si la POC sale bien.
  • Es prueba de concepto. El stack es deliberadamente sencillo (Vite + React + JS plano + CSS plano + Open-Meteo sin API key) para minimizar fricción.
Mensaje de apertura

"Hasta ahora Claude os ha ayudado a escribir documentación. Hoy os va a construir software ejecutable. La diferencia es la misma que entre un escritor y un programador — y vais a ver que con Claude no necesitáis distinguir entre los dos."

01

Qué quiero que se lleven

3 cosas, no más

  1. Que sepan crear un proyecto React desde cero pidiéndoselo a Claude. No memorizando comandos npm, sino sabiendo cómo encargar un proyecto.
  2. Que entiendan el ciclo "prompt → revisar → iterar". Claude no acierta a la primera. Lo importante es saber leer su salida y corregir.
  3. Que cada uno tenga su weather app corriendo en su localhost:5173 al final. Es el "trofeo" — verla pintando datos reales de cualquier ciudad del mundo.

Qué NO quiero hacer hoy

  • No Vercel ni deploy. La app vive en localhost. Punto.
  • No entrar en CSS pixel-perfect. Si Claude lo deja "feo pero funcional", suficiente.
  • No explicar React en profundidad. La app funciona. Si quieren entender hooks, eso lo ven en otro contexto.
  • No meter TypeScript ni Tailwind. JavaScript plano + CSS plano. Reduce fricción para los no técnicos.
Trampa habitual: querer "explicar el código" cuando Claude lo genera. Si el alumno no lo pide, no lo explicas. La curiosidad se la guardan ellos.

B3

P1 — Crear la app entera con un prompt

0:30 → 1:00 · 30 min

Este es el momento clave de la sesión. Pegan el prompt, ven cómo Claude crea archivos durante varios minutos, y al final tienen una app que funciona en localhost.

0:30P1 — Scaffold con datos mock⭐ SCAFFOLD

Qué hace este prompt

  • Crea un proyecto Vite + React desde cero en la carpeta actual del alumno.
  • Genera los componentes SearchBar, WeatherCard, ForecastList.
  • Usa datos hardcodeados de Madrid, Barcelona y Buenos Aires (no toca API).
  • CSS plano. Sin Tailwind, sin TypeScript.
  • Deja el proyecto listo para npm install + npm run dev.
P1 Crear la app con datos mock
Crea una aplicación React + Vite en la carpeta actual. La carpeta ya existe y está vacía. NO crees una subcarpeta — todo va directamente aquí. REQUISITOS DE STACK: - Vite con plantilla React (JavaScript plano, NO TypeScript). - React 18. - CSS plano. NO uses Tailwind, ni styled-components, ni librerías de UI. - NO añadas axios, lodash, ni ninguna dependencia que no venga con Vite. ESTRUCTURA DE ARCHIVOS (créalos todos): - index.html - package.json (con scripts dev, build, preview) - vite.config.js - src/main.jsx (punto de entrada) - src/App.jsx (componente raíz) - src/App.css (estilos globales) - src/components/SearchBar.jsx - src/components/WeatherCard.jsx - src/components/ForecastList.jsx - src/api/weather.js - README.md QUÉ HACE CADA COMPONENTE: - SearchBar: input controlado para escribir una ciudad y un botón "Buscar". Cuando el usuario submitea, llama a una prop onSearch(city). - WeatherCard: tarjeta con ciudad, país, temperatura actual, descripción, humedad, viento. - ForecastList: lista vertical de 7 días. Cada fila: día de la semana en español, emoji del clima, máxima y mínima. DATOS MOCK (en src/api/weather.js): Exporta una función getWeatherByCity(city) que devuelva una promesa. Si la ciudad es "Madrid", "Barcelona" o "Buenos Aires" (case-insensitive), devuelve datos hardcodeados distintos para cada una. Si es otra cualquiera, también devuelve datos para que la app se vea bien — pero hardcodeados. Estructura del objeto devuelto: { city: "Madrid", country: "España", temperature: 22, description: "Soleado", humidity: 45, windspeed: 12, forecast: [ { day: "Lun", code: 0, max: 24, min: 14 }, { day: "Mar", code: 1, max: 23, min: 13 }, ... 7 días en total ] } ESTILOS: - Diseño limpio y centrado. Fondo claro. Bordes suaves redondeados. - Paleta azul/blanco. Tipografía sans-serif del sistema. - Responsive: en móvil (max-width: 600px) que se vea apilado y legible. - Si quieres CSS por componente, usa archivos .css importados desde el .jsx. REGLAS DURAS: - NO ejecutes npm install. Solo deja el proyecto listo. - NO llames a ninguna API real todavía. Mock siempre. - NO uses console.log para debug. Si necesitas log, déjalo solo en errores. - Comentarios mínimos. Solo donde el "por qué" no sea obvio. - README en español, corto: cómo arrancar (npm install, npm run dev) y qué hace cada componente en una línea.

Qué pasa después de pegar P1

  1. Claude empieza a crear archivos. Verás varias acciones de "Write" en el panel.
  2. Al terminar, te muestra el árbol de archivos creados.
  3. Tú ejecutas en terminal:
    • npm install (tarda ~30 seg)
    • npm run dev
  4. Vite arranca en http://localhost:5173. Lo abres en el navegador.
  5. Si funciona — siguiente bloque. Si no — saltas a Troubleshooting.
Cómo enseñarlo

Mientras Claude trabaja (tarda 1-2 minutos), no te quedes en silencio. Explica qué está pasando: "Claude está creando los archivos uno a uno. Es como si le hubieras pedido a un becario que monte el proyecto, pero el becario tarda 90 segundos en vez de 4 horas."

Cuando salga el árbol de archivos, abre cada uno en VS Code y muéstralo 10 segundos. Sin leer el código en voz alta. Solo "esto es lo que pinta la tarjeta", "esto es la barra de búsqueda".

Si Claude propone Tailwind o TypeScript: respondes "no, JavaScript plano y CSS plano", y reintenta. No discutas el por qué con la clase — luego en privado les explicas que Tailwind hubiera complicado el setup.

P2 — Conectar a Open-Meteo

1:20 → 1:55 · 35 min

El bloque "wow". Pasamos de datos inventados a datos reales del clima del momento. Y lo hacemos sin escribir una línea de código.

1:20P2 — Sustituir mocks por API real⭐ API REAL
P2 Conectar a Open-Meteo (sin API key)
La app ya funciona con datos mock. Ahora conéctala a la API real de Open-Meteo. Open-Meteo es gratis y NO requiere API key. NECESITAS DOS LLAMADAS: 1. GEOCODING — convertir el nombre de la ciudad a coordenadas: GET https://geocoding-api.open-meteo.com/v1/search?name=Madrid&count=1&language=es&format=json Respuesta de ejemplo: { "results": [ { "name": "Madrid", "latitude": 40.4165, "longitude": -3.70256, "country": "España" } ] } Si results está vacío o no existe → ciudad no encontrada. 2. CLIMA + FORECAST — con esas coordenadas: GET https://api.open-meteo.com/v1/forecast?latitude=40.4&longitude=-3.7¤t_weather=true&hourly=relative_humidity_2m&daily=temperature_2m_max,temperature_2m_min,weathercode&timezone=auto De la respuesta saca: - current_weather.temperature - current_weather.windspeed - current_weather.weathercode - hourly.relative_humidity_2m[0] (humedad de la hora actual) - daily.time[] (fechas de los próximos 7 días) - daily.temperature_2m_max[] y temperature_2m_min[] - daily.weathercode[] CAMBIOS A HACER: 1. Modifica src/api/weather.js. Reemplaza getWeatherByCity(city) por una versión que: - Llame primero al endpoint de geocoding. - Si no devuelve resultados, lance: throw new Error("Ciudad no encontrada"). - Si lo devuelve, llame al endpoint de forecast con esas coordenadas. - Devuelva un objeto normalizado con la misma forma que ya usaba la app: { city, country, temperature, description, humidity, windspeed, forecast } - El array forecast tiene 7 entradas: { day, code, max, min }. "day" es el día de la semana en español ("Lun", "Mar", "Mié", etc.). 2. Mapea el weathercode a una descripción en español. Crea una función weatherCodeToText(code) en el mismo archivo: - 0: "Despejado" - 1, 2, 3: "Parcialmente nublado" - 45, 48: "Niebla" - 51, 53, 55: "Llovizna" - 61, 63, 65: "Lluvia" - 71, 73, 75: "Nieve" - 80, 81, 82: "Chubascos" - 95, 96, 99: "Tormenta" - cualquier otro: "Variable" 3. La firma pública de getWeatherByCity(city) NO cambia. App.jsx ya la usa, no debería tener que tocarse. REGLAS DURAS: - Usa fetch nativo, NO axios ni librerías HTTP. - NO uses variables de entorno ni archivos .env. Esta API no necesita key. - NO añadas dependencias nuevas al package.json. - Si una llamada falla por red, el error que lance debe propagarse para que el componente lo capture.

Cómo verificar P2

  1. Vite recarga automáticamente — la app sigue abierta en localhost:5173.
  2. Escribes "Lima" → datos reales del clima de Lima ahora mismo.
  3. Escribes "Sevilla" → datos reales de Sevilla.
  4. Escribes "asdfg" → la consola del navegador (F12) muestra "Ciudad no encontrada". La UI puede romperse — eso lo arreglamos en P3.
  5. Mensaje a la clase: "Ese número que sale ahora no me lo he inventado. Es lo que está pasando ahora mismo en la calle de Sevilla."
Cómo enseñarlo

Antes de pegar P2, abre la URL del geocoding directamente en una pestaña del navegador (https://geocoding-api.open-meteo.com/v1/search?name=Madrid&count=1&language=es). Enseña el JSON crudo. "Esto es lo que va a recibir Claude. No magia — JSON. Cualquiera puede leerlo."

1:55P3 — Estados, errores, forecast 7 días⭐ PULIDO
P3 Pulido: estados, errores, forecast, responsive
La app ya conecta a Open-Meteo. Ahora púlela: 1. ESTADO DE CARGA: - En App.jsx, añade un estado loading (boolean). - Pónlo a true antes de cada llamada, false al terminar (éxito o error). - Mientras loading sea true, muestra un texto "Cargando…" o un spinner CSS simple en lugar de la WeatherCard. 2. ESTADO DE ERROR: - Añade un estado error (string o null). - En el catch del try/catch que llama a getWeatherByCity, guarda el mensaje del error. - Cuando error tenga valor, muestra una caja con fondo rojo claro, borde rojo, y el mensaje. Debajo, un botón "Reintentar" que vuelve a ejecutar la última búsqueda. - Si error está activo, NO muestres la WeatherCard ni la ForecastList. 3. FORECAST 7 DÍAS: - La ForecastList ya recibe el array forecast. Asegúrate de que pinta los 7 días en columnas/filas legibles. - Cada fila: día de la semana, emoji según el weathercode, máxima/mínima. - Mapeo emoji recomendado: 0: ☀️ 1, 2, 3: ⛅ 45, 48: 🌫️ 51-55: 🌦️ 61-65, 80-82: 🌧️ 71-75: ❄️ 95-99: ⛈️ otros: 🌡️ 4. BÚSQUEDA POR DEFECTO: - Al cargar la página por primera vez, llama automáticamente a getWeatherByCity("Madrid") para que se vea contenido sin tener que escribir nada. 5. CSS Y RESPONSIVE: - La WeatherCard centrada horizontalmente, con max-width 480px. - Tipografía clara, jerarquía visual: temperatura grande, descripción mediana, datos secundarios pequeños. - En móvil (max-width: 600px), la ForecastList que se vea apilada en vertical sin overflow horizontal. - Pequeña transición de opacidad (200ms) cuando cambia la WeatherCard. 6. README.md: - Actualízalo con: cómo arrancar, qué API usa y por qué (sin key), estructura de archivos en una línea cada uno. REGLAS DURAS: - Sigue sin librerías de UI. CSS plano. - NO añadas localStorage ni geolocalización (eso queda como reto opcional para los que terminen pronto). - NO añadas tests. - Comentarios mínimos.

Cómo verificar P3

B6

Refuerzo y cierre

2:40 → 3:00 · 20 min

No todos van a llegar a P3 al mismo tiempo. Aquí gestionamos las velocidades distintas y cerramos.

Cómo orquestar la diferencia de ritmos

  • Quien terminó P3 antes: dale uno de los retos opcionales (lista abajo). Pueden hacerlo solos con un prompt corto a Claude.
  • Quien sigue en P2: sentarte 5 min con esa persona. Probablemente la respuesta de la API no encaja con lo que su App.jsx espera. Abre F12 → Network → Response y mira el JSON real.
  • Quien sigue en P1: revisar si Claude metió Tailwind o TypeScript a pesar de las reglas. Si sí, "tira y empieza de nuevo" es válido — son 2 minutos.

Retos opcionales (compartir con los rápidos)

  • 📍 Mi ubicación: botón que use navigator.geolocation para obtener lat/lon y consultar Open-Meteo directo (sin geocoding).
  • 💾 Última búsqueda: persistir la última ciudad en localStorage.
  • 🌙 Modo oscuro: toggle de tema usando prefers-color-scheme.
  • 🕘 Histórico: chips clickables debajo de la SearchBar con las últimas 5 ciudades buscadas.
  • 🌡️ Unidades: toggle °C / °F (Open-Meteo soporta el parámetro temperature_unit).

Cierre — los últimos 5 minutos

  1. Cada alumno enseña su pantalla al resto: escribe el nombre de su pueblo o barrio en su app.
  2. Aplauso colectivo cuando salen los datos reales.
  3. Comando de commit + PR (que no quede sin commitear lo que han hecho): 
    git add alumnos/NOMBRE/weather-report/
git commit -m "feat(NOMBRE): weather app funcional"
git push -u origin feature/weather-NOMBRE
gh pr create --repo Power-Fox-Solutions/powerfox-formacion-claude --base main --head USUARIO:feature/weather-NOMBRE --fill
Frase de cierre

"Esa app no la has escrito tú. Pero la has dirigido tú. La diferencia entre escribir código y dirigir a quien lo escribe es la misma que entre tocar un instrumento y dirigir una orquesta. Hoy habéis dirigido."

Si la POC va bien: en una sesión futura subimos la app a Vercel. Cada alumno tendría una URL pública. Si la sesión se complica hoy, mejor consolidar localhost.

Si algo se rompe

Lo que va a pasar y cómo salir rápido.

SíntomaCausa probableSolución rápida
npm install falla Node desactualizado node --version debe ser 20+. Si no, descargar de nodejs.org.
Vite no arranca Puerto 5173 ocupado Vite saltará a 5174 solo. Si no, npx kill-port 5173.
Página en blanco Error en consola JS F12 → Console. Copiar el error y pegárselo a Claude: "tengo este error: [pegar]".
API no responde CORS o ciudad inexistente Probar la URL de la API directo en navegador. Si responde JSON, la app está mal. Si no, ciudad no encontrada.
Claude crea Tailwind aunque dije que no Ignoró parte del prompt "Quita Tailwind. Usa CSS plano."
Hot reload no funciona Vite cacheado Ctrl+C en terminal. npm run dev de nuevo.
Alumno trabajó fuera de su carpeta Pegó P1 sin estar en alumnos/NOMBRE/weather-report/ git status → ver qué archivos creó. git restore --staged . → mover archivos a la carpeta correcta o tirar y empezar.
Alumno totalmente perdido Saltó un paso Le pides que haga pwd y git status. Le dices qué carpeta abrir. Vuelves al timeline.
Regla de oro: si Claude propone una solución que no entiendes en clase, no la apliques. Le dices "espera, hazlo más simple". Mejor algo que funciona aunque sea feo.
