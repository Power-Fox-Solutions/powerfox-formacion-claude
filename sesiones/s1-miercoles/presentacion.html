¿Por qué importa a todos? El gerente entiende por qué un cambio no aparece al día siguiente. El usuario sabe que en TEST puede probar sin miedo. El desarrollador tiene un proceso claro.

🌍 ¿Por qué múltiples entornos?

  • Aislamiento de riesgo: un error en DEV nunca toca PROD
  • Pruebas reales sin afectar datos de producción
  • Control de versiones: saber qué versión tiene cada entorno
  • Gobernanza: solo administradores despliegan a PROD

💡 Mínimo recomendado: 3 entornos (DEV + TEST + PROD). Ideal: añadir UAT para validación con usuario final.

📦 Managed vs Unmanaged

AspectoUnmanagedManaged
¿Dónde?Solo en DEVTEST, UAT, PROD
¿Editable?SíNo (bloqueada)
¿Eliminar?Componente por componenteSe elimina todo junto
¿Versionado?ManualAutomático con import

⚠️ NUNCA soluciones unmanaged en producción. Si necesitas un hotfix, hazlo en DEV y promuévelo como managed.

🔄 GitHub/DevOps como hub central

  • Repositorio Git como fuente de verdad de soluciones
  • Branching: main (PROD), develop (DEV), feature branches
  • Pull Requests con revisión antes de merge
  • PAC CLI para export/import automatizado

🚀 Pipeline de despliegue

  • 1. Desarrollar en DEV (unmanaged)
  • 2. Exportar con PAC CLI → commit en Git
  • 3. Build solución managed en pipeline CI
  • 4. Deploy a TEST automáticamente
  • 5. Validar con pruebas
  • 6. Aprobar y promover a PROD

Connection References: Permiten que los flujos apunten a conexiones diferentes en cada entorno sin modificar el flujo.

Diseño de Tablas, Relaciones y Herencia

30 min · CLAVE

📐 Anatomía de una tabla Dataverse

  • Primary Name — Lo que se muestra en lookups. Debe ser descriptivo (ej: "PRY-001 - Nombre").
  • Primary Key — GUID generado automáticamente.
  • Owner — Usuario o Team propietario (define seguridad).
  • Status / Status Reason — Ciclo de vida (Active/Inactive + custom).
  • Created On / Modified On — Auditoría automática.

💡 El Primary Name es fundamental para la UX. Si es un GUID o un número, los lookups son ilegibles.

🧱 Columnas avanzadas

  • Formula: Calcula en tiempo real (🔒 solo lectura). No consume storage.
  • Calculated: Se recalcula cada 24h o al leer. Para cálculos simples.
  • Rollup: Agrega datos de hijos. Se recalcula cada hora. Ejemplo: suma de horas.
  • Auto-number: Secuencias automáticas (PRY-{SEQNUM:5}). Ideal para códigos.
  • Polymorphic Lookup: Customer field (Account o Contact) y Regarding.

🔗 Herencia de datos padre-hija — ¿Cuál elegir?

Método¿Funciona desde API?¿Tiempo real?ComplejidadCuándo usarlo
Mapped Fields❌ Solo UI✅BajaDatos que solo se crean desde formulario
Business Rule❌ Solo Form✅BajaLógica simple de formulario
Power Automate✅⚡ 5-30s delayMediaDatos informativos, tolerancia a delay
Plugin✅✅ InmediatoAltaDatos críticos, consistencia total

Regla práctica: Si el dato TIENE que estar ahí siempre, usa Plugin. Si puede esperar unos segundos, Power Automate es suficiente y más fácil de mantener.

🔗 Herencia en nuestro sistema

  • Cuando se crea una pfx_project_activity dentro de un pfx_project, hereda: moneda, cliente, y datos del proyecto padre
  • Los campos pfx_offer_created y pfx_teams_created son flags que controlan si ya se ejecutó la herencia
  • Las fórmulas de margen (🔒) se calculan automáticamente — no dependen de herencia, se recalculan en tiempo real

📝 Buenas prácticas de modelado low-code

  • 1 tabla = 1 concepto de negocio. No mezcles "Proyecto" y "Tarea" en la misma tabla.
  • Normalizar, pero no demasiado. Dataverse no es SQL puro — desnormalizar lookup names es aceptable.
  • Status Reason personalizado para tu flujo de negocio (Borrador → Enviado → Aprobado → Cerrado).
  • Alternate Keys para claves de negocio (código de proyecto, NIF de empresa).
  • Auditoría habilitada en tablas con datos sensibles o regulados.

Formularios Model-Driven: Tips Avanzados + Vistas

25 min

📝 Tips avanzados de formularios

  • Formularios por rol: Usa Form Selector para mostrar formularios diferentes según el Security Role del usuario
  • Quick View Forms: Muestran datos del padre en el hijo sin JavaScript — muy útil para ver info del Proyecto desde una Actividad
  • Card Forms: Vista compacta para vistas de tarjetas
  • Business Rules en formulario: Lógica sin código para show/hide, set required, set default
  • Tabs con "expand by default: No": Mejoran rendimiento — no cargan sus controles hasta que se abren

💡 Crea formularios personalizados para tu solución en vez de editar los nativos. Evita conflictos con otros desarrollos.

👁️ Vistas y Subgrids

  • System Views — Compartidas por todos. Personal Views solo para el usuario.
  • Lookup Views — Controla qué columnas ve el usuario al buscar en un lookup.
  • Quick Find View — Define qué campos son buscables. Impacta rendimiento.
  • Subgrids: Muestran registros relacionados. Editable grids para inline editing.
  • Dataverse Search: Alternativa a Quick Find para tablas grandes (>100K registros).

⚠️ Quick Find busca con LIKE %texto%. En tablas grandes es lento. Usa Dataverse Search en su lugar.

FetchXML Builder — Demo en Vivo

20 min

⭐ ¿Qué es FetchXML Builder?

La herramienta más usada de XRMToolBox. Permite:

  • Construir consultas arrastrando tablas y condiciones
  • Probar ejecutando contra el entorno real
  • Copiar FetchXML para usar en vistas, reportes, plugins o JavaScript
  • Convertir a Web API URL ($filter, $select, $expand)
  • Hacer joins entre tablas con link-entity
  • Funciones aggregate (count, sum, avg) para reportes rápidos

💡 Para todos los perfiles

  • Gerente: "¿Cuántos proyectos activos tiene cada coordinador?" — FetchXML con aggregate y groupby, resultado en 2 minutos.
  • Usuario: "¿Qué proyectos no tienen horas registradas este mes?" — Consulta con left join y condición null.
  • Técnico: "Dame los campos de pfx_project_activity que son lookup" — Metadata query directa.

No necesitas saber código. FetchXML Builder es visual — arrastras, configuras, ejecutas. Si luego necesitas el XML o la Web API URL, lo copias.

Introducción a JavaScript en Dynamics 365

25 min

⚡ Los 3 eventos principales

EventoCuándoUso típico
OnLoadAl abrir el formularioInicializar, ocultar campos, validar permisos
OnChangeAl cambiar un campoCalcular, auto-rellenar, filtrar lookups
OnSaveAntes de guardarValidar datos, prevenir guardado si error

⚠️ Xrm.Page está deprecado desde v9. Siempre usa formContext obtenido de executionContext.getFormContext().

🔧 formContext — Lo básico

// Obtener formContext (SIEMPRE así) function onLoad(executionContext) { var fc = executionContext.getFormContext(); // Leer un valor var nombre = fc.getAttribute("pfx_name").getValue(); // Escribir un valor fc.getAttribute("pfx_name").setValue("Nuevo"); // Ocultar un campo fc.getControl("pfx_campo").setVisible(false); // Hacer obligatorio fc.getAttribute("pfx_email") .setRequiredLevel("required"); }

📐 Estructura recomendada: Namespace

var LITOCLEAN = LITOCLEAN || {}; LITOCLEAN.Proyecto = (function() { function onLoad(executionContext) { var fc = executionContext.getFormContext(); // Tu lógica aquí } function onChange_Estado(executionContext) { var fc = executionContext.getFormContext(); // Reaccionar al cambio } return { onLoad: onLoad, onChange_Estado: onChange_Estado }; })();

💡 Un archivo JS por entidad. Namespace para evitar colisiones. Solo expón las funciones que van en los event handlers.

🔗 Cómo vincular JS a un formulario

  • 1. Crear Web Resource tipo JavaScript (.js)
  • 2. Abrir el formulario en el editor
  • 3. Form Properties → Add Library → seleccionar tu JS
  • 4. Event Handlers → Add → elegir función (ej: LITOCLEAN.Proyecto.onLoad)
  • 5. Marcar ✅ "Pass execution context as first parameter"
  • 6. Save + Publish

⚠️ Si olvidas marcar "Pass execution context", el formContext será undefined y el script fallará silenciosamente.
