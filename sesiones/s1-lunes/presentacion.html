🏗️ Los 5 pilares

  • Power Apps — Model-Driven y Canvas para UI. Model-Driven sigue la estructura de Dataverse; Canvas da control total de diseño.
  • Power Automate — Flujos cloud y desktop. Triggers de Dataverse, conectores a 1000+ servicios.
  • Power BI — Análisis y visualización conectado directamente a Dataverse.
  • Copilot Studio — Agentes IA con acceso a datos empresariales.
  • Power Pages — Portales web sobre Dataverse para usuarios externos.

🤖 Claude Code como asistente IA

Claude Code se integra en el terminal y VS Code para:

  • Generar JavaScript para formularios Model-Driven
  • Crear consultas FetchXML y Web API
  • Depurar errores y sugerir patrones
  • Documentar código y soluciones existentes
  • Acelerar tareas repetitivas (CRUD, validaciones)

Dato clave: En un proyecto real, usar IA como asistente redujo entre un 60-70% el tiempo de desarrollo en Canvas Apps.

Modelo de Datos: Tablas y Relaciones

45 min · CLAVE

🗄️ Tipos de tablas en Dataverse

  • Standard — Tablas predefinidas (Account, Contact, etc.)
  • Custom — Creadas para tu solución (prefijo: pfx_*)
  • Virtual — Conectan datos externos sin migrarlos
  • Elastic — Para grandes volúmenes (basadas en Cosmos DB)

💡 Tip: Siempre usa un publisher con prefijo corto y significativo. Evita el prefijo "new_" por defecto.

🔗 Tipos de relaciones

  • 1:N (One-to-Many) — La más común. Ej: Proyecto → Actividades. El hijo tiene un Lookup al padre.
  • N:N (Many-to-Many) — Tabla intermedia automática. Ej: Contacto ↔ Habilidades.
  • 1:1 (One-to-One) — Simulada con Lookup único. Ej: Contacto → Detalle.

⚠️ Cuidado: El comportamiento en cascada (Assign, Delete, Reparent) tiene impacto directo en rendimiento y datos. Configúralo conscientemente.

📝 Buenas prácticas de modelado

  • Nombra las tablas en singular (pfx_project, no pfx_projects)
  • Usa Display Name descriptivo y Schema Name técnico
  • Evita campos "por si acaso" — cada columna tiene coste
  • Documenta las relaciones y su comportamiento de cascada
  • Usa Alternate Keys para integraciones externas
  • Prefiere Lookups sobre campos de texto libre cuando hay referencia
  • Campos calculados y rollup: no consumen API calls pero se recalculan asíncronamente
  • Campos fórmula (🔒): se calculan en tiempo real, son de solo lectura vía API

🔐 Security Roles

Cada rol define permisos CRUD+A (Create, Read, Write, Delete, Append, Append To, Share, Assign) por tabla y nivel de acceso:

  • User — Solo sus registros
  • Business Unit — Registros de su BU
  • Parent: Child BU — Su BU + las hijas
  • Organization — Todos los registros

Regla de oro: Principio de mínimo privilegio. Empieza restringido y abre acceso según necesidad real.

👥 Teams

Tipos de Teams en Dataverse:

  • Owner Team — Puede ser propietario de registros. Comparte roles con sus miembros.
  • Access Team — No tiene roles propios. Da acceso a registros específicos compartidos.
  • Azure AD Group Team — Sincronizado con grupo de Azure AD. Los roles se asignan al grupo.

💡 Best practice: Usa AAD Group Teams para sincronizar permisos con el directorio corporativo. Simplifica la administración enormemente.

🏢 Business Units

  • Cada entorno tiene un BU raíz que no se puede eliminar
  • Los BU crean particiones de datos — los usuarios de un BU no ven datos de otro (salvo rol Organization)
  • Cada usuario pertenece a exactamente 1 BU
  • Útil para separar departamentos, filiales o regiones

🛡️ Field-Level Security

  • Controla acceso a campos específicos independiente del rol
  • Se habilita campo por campo y se gestiona con Field Security Profiles
  • Casos de uso: salarios, datos médicos, información confidencial
  • Los usuarios sin perfil ven el campo vacío o bloqueado

⚠️ Field Security impacta rendimiento en queries — úsalo solo donde realmente sea necesario.

👥 Los 10 roles custom de nuestro sistema

  • PM _ Admin — Full access (Org)
  • PM _ Admin Unidad de Negocio — Full access (BU)
  • PM _ Coordinador del Proyecto — Gestiona proyectos asignados
  • PM _ Gestor del Proyecto — Similar a Coordinador (diferencias sutiles)
  • PM _ Gerente Linea de Negocio — Supervisión de BU
  • PM _ Finanzas — Acceso a datos económicos
  • PM _ Representante de Ventas — Ofertas y comercial
  • PM _ Tecnico — Sus horas y actividades propias
  • PM _ Solo Lectura — Consulta sin modificación
  • PM _ App Opener — Mínimo para abrir la app

Candidatos para Field-Level Security: pfx_estimated_sale_price, pfx_actual_cost_price — ¿un Técnico necesita ver el presupuesto?

Buenas Prácticas de Arquitectura y Escalabilidad

30 min

✅ Hacer

  • Usar soluciones gestionadas (managed) en producción
  • Separar componentes en múltiples soluciones por módulo
  • Implementar ALM completo (DEV → TEST → PROD)
  • Usar Connection References y Environment Variables
  • Documentar modelo de datos y decisiones de diseño
  • Implementar auditoría en tablas críticas
  • Planificar la estrategia de backup y recuperación

❌ Evitar

  • Desarrollar directamente en producción
  • Soluciones no gestionadas en PROD
  • Plugins síncronos innecesarios (impactan UX)
  • Campos calculados en tablas con millones de registros sin probar rendimiento
  • Hardcodear GUIDs o URLs en código
  • Un solo "super-role" que lo puede todo
  • Ignorar los límites de API: 6,000 requests / 5 min por usuario

Takeaways del Día

  • Power Platform = ecosistema integrado. Cada pieza tiene su rol; la clave es saber cuándo usar qué.
  • 4 soluciones bien separadas (Core, WebResources, Dataflows, CustomAPI) — cada capa se despliega independientemente.
  • El modelo de datos es la base de todo. Los campos 🔒 fórmula son solo lectura. El cascade behavior tiene consecuencias reales.
  • Seguridad ≠ permisos. Es una estrategia que combina roles, BUs, teams y field security.
  • 10 roles custom con niveles de acceso claros. Principio de mínimo privilegio.
  • Claude Code acelera, no reemplaza. El criterio técnico sigue siendo del desarrollador.
  • Arquitectura se piensa antes de construir. Miércoles veremos cómo llevar esto a entornos y ALM.
