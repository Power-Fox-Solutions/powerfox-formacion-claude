Secret scanning + push protection

qué credenciales detecta GitHub, cuándo te bloquea y qué hacer si saltan las alarmas

En esta sesión activamos dos capas que escanean cada commit en busca de credenciales. La primera detecta patrones conocidos (API keys de proveedores grandes). La segunda rechaza el push antes de que la credencial llegue al repo público. Es una red de seguridad adicional sobre la regla social de CLAUDE.md: funciona aunque el alumno nunca la haya leído.

Lo que detecta vs. lo que no

Sí detecta
Credenciales con patrón reconocido
  • sk- de OpenAI / Anthropic
  • GitHub tokens ghp_, gho_, ghs_
  • AWS Access Key ID + Secret
  • Stripe, Slack, Google Cloud, Azure keys
  • SSH private keys (bloques BEGIN PRIVATE KEY)
  • Connection strings de Mongo, Postgres, Redis con password
No detecta
Secretos genéricos sin patrón
  • Contraseñas en texto plano password = "abc123"
  • Secretos custom de Power Fox BI sin firma pública
  • Variables .env de clientes internos
  • GUIDs de Dataverse o IDs de registro
  • URLs internas tipo acme.internal
  • Nombres de clientes reales (privacidad, no credencial)
Importante
Push protection NO sustituye el sentido común. Solo cubre los patrones de la lista de proveedores soportados. El resto (datos de clientes, URLs internas, passwords custom) depende de la disciplina y del checklist del PR template.

Cómo salir del bloqueo sin romper nada

01
Lee el mensaje
GitHub te dice archivo, línea y tipo de token. Toda la info está ahí.
# salida típica remote: - Push cannot contain secrets remote: OpenAI API Key remote: src/config.py:12
02
Quita el secreto del commit
Editar el archivo encima no basta — el secret sigue en el historial. Amend o reescribe.
# último commit # reemplaza la key por # os.environ["OPENAI_API_KEY"] git add . git commit --amend --no-edit
03
Push otra vez
Tras amend el hash cambió, necesitas --force-with-lease. Solo en tu rama feature, nunca en main.
git push --force-with-lease
04
Rota la key igualmente
Aunque GitHub la bloqueó, pudo quedar en cache, portapapeles, historial shell o logs. Considérala comprometida.
# OpenAI: platform.openai.com # /api-keys # GitHub: settings/tokens # AWS: IAM / credentials
Edge case · falso positivo
Raro. Solo si la string parece una key pero no lo es (placeholder en un test, hash público). En ese caso sigue la URL de bypass del error. En este repo la política es: no bypass — renombra el placeholder (sk-EJEMPLO, YOUR_KEY_HERE) para que no encaje con el patrón del proveedor, y pushea otra vez.
Edge case · secreto en commit antiguo
Si el secret está antes del último commit, amend no basta. Opciones: git rebase -i HEAD~N para reescribir el histórico (solo antes de que nadie lo haya hecho pull), o crear un commit nuevo fix(security): reemplaza credencial con placeholder. En los dos casos: rota la original igualmente.

Para no olvidar

Regla de oro
Si una clave tocó el working directory, asume que está comprometida. Rotarla siempre. El bloqueo de GitHub evita que llegue al repo público, pero no evita que haya quedado en tu historial de shell, en el portapapeles, o en un log.
Cómo evitarlo en primer lugar
Nunca escribas claves en código. Usa variables de entorno (os.environ["X"], process.env.X) y un archivo .env local — que el .gitignore del repo ya excluye. Así el código va al repo y las claves se quedan en tu máquina.
La regla social sigue primero
Secret scanning es un seguro, no un sustituto de CLAUDE.md. El checklist del PR template te obliga a confirmar que no subes credenciales ni datos de cliente. La capa técnica solo corrige lo que el humano olvidó — pero lo que no tiene patrón conocido (URLs, nombres, GUIDs) pasa sin rechistar.
Nunca fuerces el bypass "para salir del paso"
Si te sale el error y no entiendes por qué, pregúntale a Yeriel antes de pulsar bypass. Un bypass mal dado publica una credencial real en un repo público — difícil de limpiar, fácil de explotar.
Qué hacer si ya se coló algo antes de todo esto
El scanning mira commits nuevos. Si una credencial vieja ya está en el histórico de main, este toggle no la limpia. Para eso: rotar la key en el proveedor y abrir incidente con Yeriel — borrar del histórico es posible pero doloroso y no evita que alguien ya la tenga.
