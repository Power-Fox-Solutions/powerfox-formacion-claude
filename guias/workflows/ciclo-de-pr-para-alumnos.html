Ciclo completo de una Pull Request

Del git push al merge,
qué pasa ahora paso a paso

Lo que ve un alumno, lo que recibe Yeriel y qué automatismos saltan en cada fase.

El ciclo de PR en este repo dejó de ser "abro, espero, alguien aprueba, mergeo". Hoy hay templates que te guían, linters que te corrigen, bots que saludan, etiquetas que se asignan solas y reglas de merge que limpian ramas sin intervención. Esta guía cubre qué cambió en la sesión del 23-abr-2026 — PRs #3 y #4 mergeadas, más ajustes del repo aplicados vía API.

Quién hace qué

Tres actores, tareas bien separadas

Alumno
Abre la PR y corrige lo que el CI le marque. Crea rama, edita su carpeta, pushea, abre PR con gh pr create. A partir de ahora el template le guía con checklist de privacidad. Si commitlint falla, corrige y pushea otra vez.
GitHub
Ejecuta los automatismos antes de que Yeriel mire nada. Rellena el template, lanza commitlint, dispara welcome bot, consulta CODEOWNERS y asigna reviewer, borra la rama tras merge.
Yeriel
Solo mira PRs que pasan la criba automática. Recibe notificación cuando le asignan como reviewer (zona sensible). Revisa, comenta si hace falta y hace gh pr merge N --squash. No necesita borrar ramas ni elegir tipo de merge — está configurado.
Comparativa

Qué cambia momento a momento

Momento Antes Ahora
Al abrir una PR Descripción vacía, cada uno inventa formato Checklist pre-rellenado: privacidad, convención de commits, scope de carpeta
Al abrir un issue Issue en blanco, sin estructura Dos plantillas: Duda de alumno y Bug o error en el repo. El issue en blanco está desactivado
Commit con mensaje no-convencional Pasa, se mergea igual Workflow commitlint marca la PR en rojo hasta corregir
Primera PR o issue de cada alumno Silencio absoluto hasta que Yeriel responde Welcome bot publica automáticamente un saludo con los pasos clave
PR que toca zonas sensibles Yeriel tiene que acordarse de auto-asignarse CODEOWNERS lo auto-asigna en /.github/, /sesiones/, /demos/, /prompts/, /.claude/ y raíz
PR que toca solo /alumnos/[nombre]/ Espera aprobación obligatoria Sin code owner, 0 aprobaciones requeridas → Yeriel puede mergear directamente sin gate
Botón de merge Tres opciones: merge commit / squash / rebase Solo squash and merge. Historial de main limpio, un commit por PR
Tras mergear La rama seguía ahí, se acumulaban Rama origen autoborrada en GitHub. git pull --prune la limpia en local
Estilo entre alumnos CRLF vs LF, indentación a ojo .editorconfig unifica encoding UTF-8, LF y 2 espacios de indent
Todo el trabajo mecánico lo hace el repo. Yeriel solo decide si el contenido vale.
Qué ve el alumno que antes no veía

Cinco señales nuevas en su experiencia

Al ejecutar gh pr create
Cuerpo de PR pre-rellenado
El checklist aparece solo. Si el alumno lo usa --fill, se queda el template; si usa --body, lo sobreescribe. Lo recomendable: editar el template.
A los 20-60 segundos
Check de commitlint
Aparece un check en la pestaña "Checks" de la PR. Si falla, abre el log y verá qué commit no cumple Conventional Commits.
En primera interacción
Comentario de bienvenida
El welcome bot deja un comentario automático en la PR o issue con los pasos clave y el checklist. Solo dispara una vez por persona, nunca repite.
Si toca zona sensible
Yeriel aparece en "Reviewers"
Caja "Reviewers" a la derecha con YerielMangin ya marcado. El alumno no tiene que mencionarlo — CODEOWNERS lo hizo por él.
Al intentar commit con secret
Push rechazado
Si mete sin querer una API key o token conocido, GitHub le devuelve error antes de publicar. Detalles en la guía security/escaneo-de-secretos.html.
Flujo real en terminal

Un ejemplo completo, de rama nueva a main sincronizada

# 1. Parte limpia desde main actualizado
git switch main && git pull

# 2. Rama con nombre convencional (feature/..., fix/..., docs/...)
git switch -c feature/alumno-demo-pregunta

# 3. Edita dentro de /alumnos/[tu-nombre]/ y commit con Conventional Commits
git add alumnos/tu-nombre/
git commit -m "feat(alumnos): añade demo de pregunta para sesión 3"

# 4. Push y PR — el template aparece solo al abrir
git push -u origin HEAD
gh pr create

# 5. Si commitlint falla, corriges y repushea (NO abras otra PR)
git commit --amend
git push --force-with-lease

# 6. Yeriel mergea cuando apruebe; tu rama local sigue viva
git switch main && git pull --prune
git branch -D feature/alumno-demo-pregunta  # la local se borra a mano
Detalles que importan

Lo que conviene recordar sin pensar

Por qué 0 aprobaciones en main. Yeriel es la única con permiso de merge y GitHub nunca deja que el autor apruebe su propia PR. Subir el contador a 1 sólo bloquearía sus propios PRs sin aportar nada. La disciplina de revisión la mantiene ella manualmente — las reglas técnicas solo limitan el ruido.
Nunca pushees desde tu main del fork. Aunque la PR salga mergeable, rompe la convención: el nombre de rama main → main no dice nada sobre el cambio. Usa siempre feature/..., fix/... o docs/.... Si lo haces, espera un comentario amistoso de Yeriel recordándotelo.
El template de PR guía, no obliga. Puedes borrar el checklist si tu PR es trivial. Lo que sí es obligatorio: mensajes de commit en Conventional Commits (commitlint bloquea) y respeto a la privacidad del repo público (secret scanning bloquea). Todo lo demás lo valora Yeriel caso por caso.
Las reglas completas viven en CLAUDE.md. Esta guía resume el comportamiento visible. Para convenciones de estilo de contenido, scope de carpetas y qué no subir al repo público, la fuente canónica es CLAUDE.md en la raíz.
