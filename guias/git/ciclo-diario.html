El ciclo diario: alumno → PR → merge

cómo fluye el trabajo en un repo con un facilitador y varios alumnos

Cada alumno trabaja en su propia rama local tocando exclusivamente su carpeta /alumnos/[su-nombre]/. El facilitador (Yeriel) no ve nada hasta que abren PR y él la mergea. Después hace git pull en main y recibe solo lo consolidado — las ramas locales de otros alumnos no le llegan.

Actor 01
Alumno
Crea rama, edita su carpeta, pushea, abre PR.
Actor 02
GitHub (origin)
Hospeda main protegida, PRs abiertas y ramas remotas.
Actor 03
Yeriel (facilitador)
Revisa PRs, mergea, hace pull a main para ver lo nuevo.

Quién ve qué en cada momento
Momento Lo que ve el alumno Lo que ve Yeriel al hacer git pull en main
Durante su trabajo local Su rama + lo que había en main cuando la creó. No ve ramas de otros alumnos. Nada nuevo de ese alumno hasta que él haga push.
Tras el push Su rama remota existe en GitHub (origin/feature/...). Nada aún — la PR está abierta pero no mergeada.
Tras la PR abierta Ve su PR en GitHub. Puede recibir comentarios. Ve la PR con gh pr list, pero git pull en main no trae nada.
Tras el merge Su PR se marca como merged. Su rama puede autoborrarse si está configurado. Al hacer git pull en main recibe los commits del alumno consolidados.
Comandos típicos del día
01 · ALUMNO
Crear rama y trabajar
Partir siempre de main actualizado para no arrastrar versiones viejas.
git switch main && git pull git switch -c feature/mi-cambio
02 · ALUMNO
Pushear y abrir PR
El -u vincula la rama local con la remota para futuros git push sin args.
git push -u origin HEAD gh pr create --fill
03 · YERIEL
Revisar y mergear
Revisa, deja comentarios y mergea cuando esté listo. La rama se autoborra si está configurado.
gh pr list gh pr view 12 gh pr merge 12 --squash
04 · YERIEL
Recibir lo consolidado
Simple pull en main. Solo llegan commits ya mergeados, no ramas sueltas de alumnos.
git switch main git pull
¿Chocan entre sí los alumnos?
No, mientras cada uno toque solo su carpeta /alumnos/[su-nombre]/. Git mergea sin conflicto porque los archivos no se solapan. El único caso de conflicto real aparece si un alumno edita archivos fuera de su carpeta que otra PR ya tocó — por eso el CLAUDE.md lo prohíbe.
Lo que un pull NO trae
git pull en main solo descarga commits ya mergeados. No trae ramas locales de compañeros (viven solo en su máquina) ni PRs abiertas (están en GitHub pero aún no en main). Para ver PRs pendientes usa gh pr list.
