Actualizar tu rama con main — sin conflictos

qué pasa cuando Yeriel mergea PRs y tú sigues trabajando en tu rama

Mientras trabajas en tu rama, main avanza cada vez que Yeriel mergea una PR de otro compañero. En algún momento conviene traer esos cambios a tu rama. Si solo editaste tu propia carpeta /alumnos/[tu-nombre]/, es una operación invisible y sin riesgos. Esta guía cubre los tres escenarios que te puedes encontrar.

Los tres escenarios que puedes encontrarte
Escenario A · Happy path
Solo toqué mi carpeta

Es el caso normal. Git ve que tus cambios viven en archivos que nadie más editó, así que reaplica tus commits sin conflicto. Ganas: ahora tienes lo que hicieron tus compañeros en tu working directory.

git switch feature/mi-rama git fetch origin git merge origin/main # → sin conflictos
Escenario B · Trabajos a medias
Tengo cambios sin commitear

Si hay archivos modificados sin git add + commit, Git se niega a mergear. Guarda tus cambios a un lado con stash, mergea, y recupéralos.

git stash git fetch origin git merge origin/main git stash pop # recupera
Escenario C · Conflicto
Edité un archivo fuera de mi carpeta

Si tocaste, por ejemplo, /sesiones/01/… y otra PR también lo tocó, Git para el merge y te muestra CONFLICT. Toca resolver a mano. El CLAUDE.md lo prohíbe por esto mismo — si sigues la regla, este caso no aparece.

# abrir el archivo en conflicto # resolver <<<< y >>>> git add archivo-conflicto git commit
Merge vs rebase · ¿cuál usas?
git merge origin/main
git rebase origin/main
Recomendado para alumnos.
  • Crea un commit de merge que une las dos líneas de historia
  • No reescribe tus commits — lo que ya pusheaste sigue igual
  • Si hay conflicto, lo resuelves una sola vez
  • Historial con forma de Y, con ramas visibles
Para PRs más pulidas.
  • Reaplica tus commits encima del último main
  • Reescribe hashes — si ya pusheaste, requiere push --force-with-lease
  • Si hay conflictos, puede pedírtelos commit por commit
  • Historial en línea recta, más limpio para revisores
¿Cuándo actualizarse?
No es obligatorio hacerlo cada día. Basta con hacerlo antes de abrir la PR, o si GitHub te avisa "this branch is out-of-date". Si tu rama vive más de una semana, mejor actualizarla un par de veces para que los conflictos (si llegan) aparezcan en pequeño.
Nunca hagas push --force a main
El rebase reescribe commits. Si alguna vez rebase una rama ya pusheada, usa git push --force-with-lease (nunca --force a secas) y solo sobre tu propia rama de feature. Jamás fuerces push a main — está protegida y borrarías trabajo de todo el equipo.
