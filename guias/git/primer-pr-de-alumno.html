árbol de decisión para revisar sin bloquear el aprendizaje

La tentación es rechazar por "no cumple la convención" y pedir que rehaga todo. Eso frustra al alumno y rompe el momentum. La decisión correcta no es mergear o rechazar, es cómo enseñar sin bloquear el primer paso.

Caso real · Luis Hoyos · PR #2
Primer PR abierto desde main, no desde feature/...

El alumno siguió el tutorial de README, forkeó el repo, hizo un commit directo a su main y abrió PR contra el main del repo upstream. Técnicamente rompe la convención de "rama feature/descripcion", pero el cambio era limpio, tocaba solo su carpeta y no había secretos. Se mergeó igual, con comentario educativo para la próxima vez.

Los 4 checks en orden de prioridad
i
Scope — ¿su carpeta y nada más?

Es la única regla sagrada. Si un alumno tocó algo fuera de /alumnos/[su-nombre]/ (otra carpeta, sesiones, .github/) hay que bloquear y explicar por qué: protege a los demás.

ii
Privacidad — ¿secretos o clientes reales?

API keys, tokens, URLs internas, nombres de clientes. El repo es público: lo que se mergea es visible para cualquiera. Si hay filtración, bloquear y pedir limpieza.

iii
Rama — ¿viene de feature/...?

Convención, no obligación técnica. Si abrió PR desde su main (caso Luis Hoyos), mergea igual y deja comentario educativo. No rechazar por esto.

iv
Commits — ¿Conventional Commits?

feat:, docs:, fix:... Si falla, pedir ajuste del mensaje antes de mergear (es barato: git commit --amend + force push).

Para la próxima, un par de cosas que nos ahorran trabajo a todos:

1. Crea una rama de feature antes de empezar.

git switch main && git pull
git switch -c feature/mi-cambio

Así tu main local siempre refleja el repo y puedes tener varias ramas en paralelo si haces varias contribuciones.

2. El mensaje de commit sigue Conventional Commits:

feat: añade ejemplo de prompts en weather-report
docs: corrige enlace roto en README
fix: arregla indentación en script de demo

Los prefijos (feat:, docs:, fix:, chore:) hacen que el historial sea legible de un vistazo.

Cualquier duda, me pingeas por aquí o en el canal del curso.

El welcome bot te ahorra el primer comentario
En este repo hay un workflow (.github/workflows/welcome.yml) que deja un comentario automático en la primera PR de cada alumno — saludo y enlace a las convenciones. Eso cubre el 80% del mensaje educativo. Tú solo tienes que añadir lo específico del caso (rama mal creada, commit sin type, etc.).
Principio: educar mergeando, no rechazando
Rechazar una PR por detalles de convención es pedagógicamente caro — el alumno pierde 20 minutos en rehacerla y a veces abandona. Mergea salvo que haya riesgo real (scope ajeno, secreto, cliente) y enseña en el comentario. La PR #2 no es el sitio para pedir perfección.
