CODEOWNERS: avisar vs bloquear

dos modos con un checkbox de diferencia — y consecuencias muy distintas

El archivo .github/CODEOWNERS tiene dos comportamientos posibles según la configuración del repo. Uno es asignación automática (el owner se añade como reviewer, pero puede mergearse sin su aprobación). El otro es requisito de aprobación (la PR queda bloqueada hasta que el owner apruebe). La diferencia es un checkbox en los ajustes de la rama protegida — pero cambia el perfil de riesgo del repo.

Los dos modos, lado a lado
Modo 1 · suave

Asignación automática

Solo con el archivo CODEOWNERS existiendo — sin marcar nada extra en GitHub. Los owners se añaden como reviewers cuando la PR toca sus archivos.

  • Alumno abre PR que toca un archivo con owner
  • GitHub añade al owner como reviewer automáticamente
  • El owner recibe notificación
  • La PR puede mergearse sin su aprobación
Útil para Equipos con flujo ágil — el owner se entera pero no bloquea. Ideal para monitorear zonas sensibles sin frenar al resto.
Modo 2 · estricto

Aprobación obligatoria

Activando Require review from Code Owners en la branch protection rule. Entonces la PR NO puede mergearse hasta que el owner apruebe.

  • Alumno abre PR que toca un archivo con owner
  • GitHub añade al owner como reviewer automáticamente
  • La PR queda con estado "BLOCKED"
  • Nadie (ni admins por defecto) puede mergear hasta la aprobación
Útil para Zonas críticas donde nada se mergea sin revisión humana del owner: configuración, CI/CD, permisos, documentación oficial.

Sintaxis del archivo CODEOWNERS
Ejemplo comentado .github/CODEOWNERS 
# ZONA LIBRE — sin owner, cualquier colaborador puede tocar
# (no aparece en CODEOWNERS = no hay owner)
# /alumnos/  → implicito: sin owner

# CONFIGURACION DEL CURSO — solo el facilitador
/.claude/          @yeriel
CLAUDE.md          @yeriel
/.github/          @yeriel

# PRESENTACIONES — facilitador + co-instructor (si lo hubiera)
/sesiones/         @yeriel

# REGLA GENERAL — fallback para todo lo no cubierto arriba
# (se aplica SOLO si no hay match mas especifico)
*                  @yeriel
Cuándo usar cada modo
Zona del repo Riesgo de cambio sin revisión Modo recomendado
/alumnos/ Bajo — cada alumno en su carpeta, no afecta a nadie más Sin owner
/sesiones/ Medio — cambios afectan material del curso pero son reversibles Aviso
/.github/workflows/ Alto — un workflow mal escrito puede leak secretos o romper CI Bloqueo
/.claude/ Alto — skills y agents compartidos afectan a todo el curso Bloqueo
CLAUDE.md Alto — instrucciones del repo, cambio silencioso tiene radio amplio Bloqueo
README.md Medio — visible al público, pero los errores son cosméticos Aviso
Cómo activar el modo bloqueo en GitHub
Settings → Branches → Branch protection rules → main → Edit.
Marca "Require a pull request before merging" y debajo activa "Require review from Code Owners". Con eso, cualquier PR que toque un archivo listado en CODEOWNERS queda bloqueada hasta aprobación del owner.
Trampa habitual: mezclar reglas de match
CODEOWNERS aplica la última regla que hace match, no todas. Si pones * @yeriel al principio y después /alumnos/ (sin owner, intención de "zona libre"), la última regla vacía sí quita el owner. Pero si es al revés (/alumnos/ vacío primero, * @yeriel después), todos los archivos — incluidos los de /alumnos/ — quedan con Yeriel como owner. Orden importa: más específico primero, fallback al final.
